# Internationalisation (i18n)

SPIP's i18n system uses language files (PHP files returning associative arrays) organized by module.
A plugin declares its module(s) in `paquet.xml` and provides a reference language file in `lang/`.

---

## Declaring a language module

In `paquet.xml`:

```xml
<!-- plugins-dist/spip/forum/paquet.xml:16 -->
<traduire module="forum" reference="fr" gestionnaire="salvatore" />
```

This tells SPIP that:
- The main lang files are named `lang/forum_XX.php` (where `XX` is the language code)
- The reference language is `fr`
- The module is managed by `salvatore` (a trad.spip.net gestionnaire — not relevant at runtime)

A plugin can declare multiple modules; the module name should match the plugin prefix (`forum`,
`mots`, `myplugin`…).

---

## Language file format

File: `myplugin/lang/myplugin_fr.php`

```php
<?php
// This is a SPIP language file  --  Ceci est un fichier langue de SPIP
return [
    // A
    'titre_section'       => 'Mes objets',
    'aucun_objet'         => 'Aucun objet trouvé',

    // I
    'info_1_objet'        => 'Un objet',
    'info_nb_objets'      => '@nb@ objets',

    // E
    'erreur_champ_vide'   => 'Le champ @champ@ est obligatoire',
];
```

- Keys are lowercase, `snake_case`
- Variable placeholders: `@nom@` — replaced at runtime by `_T('module:cle', ['nom' => $val])`
- Section comments (`// A`, `// B`…) help translators but have no runtime effect

---

## _T() — look up a translated string

```php
// ecrire/inc/utils.php:929
_T(string $cle, array $args = [], array $options = []): string
```

`$cle` is `'module:key'` — the module prefix routes to the correct lang file.

```php
// Simple lookup
echo _T('forum:aucun_message_forum');

// With variable substitution
echo _T('forum:forum_attention_nb_caracteres_mini', ['min' => 10]);
// uses @min@ placeholder in lang file value

// Cross-module (core modules: spip, ecrire, public)
echo _T('public:mots_clefs');
echo _T('ecrire:info_modification_enregistree');
```

### Options

| Option key | Default | Meaning |
|---|---|---|
| `force` | `true` | When `true`, falls back to the key itself if no translation found; `false` returns `''` |
| `sanitize` | `true` | Whether to escape HTML in `$args` values |

```php
// Return empty string rather than the raw key when translation is missing
$label = _T('myplugin:cle_optionnelle', [], ['force' => false]);
```

### Key-only fallback

When a key has no translation (e.g., missing lang file), `_T()` with `force = true` returns the key
stripped of the module prefix, with underscores turned into spaces:
`'myplugin:ma_cle_manquante'` → `'ma cle manquante'`.

---

## _L() — interpolate variables in a string

Lower-level than `_T()`. Use when you already have the translated string and need variable
substitution:

```php
// ecrire/inc/utils.php:1002
$text = _L('Bienvenue @nom@ !', ['nom' => $visiteur]);
// Replaces @nom@ with $visiteur, HTML-escaping by default
```

This is what `_T()` calls internally after fetching the translation.

---

## lang_select() — switch language context

Use when you need to generate translated output in a language other than the current context (e.g.,
sending an email in the recipient's language):

```php
// Push a new language
lang_select('en');
$message = _T('myplugin:email_body');   // fetched in English

// Pop back to previous language
lang_select(null);
```

`lang_select($lang)` pushes the current language onto a stack and switches to `$lang`.
`lang_select(null)` pops the stack and restores the previous language.

## paquet-module lang file

A second lang file type holds strings for the plugin's own description in the plugin manager UI:

File: `myplugin/lang/paquet-myplugin_fr.php`

```php
// plugins-dist/spip/forum/lang/paquet-forum_fr.php
return [
    'forum_description' => 'Forum de SPIP (privé et public)',
    'forum_slogan'      => 'Gestion des forums privés et publics dans SPIP',
];
```

Mandatory keys:
- `{prefix}_description` — shown in the plugin description panel
- `{prefix}_slogan` — one-line tagline

These strings are looked up under a special `paquet-myplugin` module (prefixed with `paquet-`),
not under the main `myplugin` module.

---

## Module naming summary

| File | Module key in `_T()` | Purpose |
|---|---|---|
| `lang/myplugin_fr.php` | `myplugin:` | Main plugin strings |
| `lang/paquet-myplugin_fr.php` | `paquet-myplugin:` | Plugin manager metadata |
| `lang/spip_fr.php` (core) | `spip:` | Core editorial strings |
| `lang/ecrire_fr.php` (core) | `ecrire:` | Backoffice strings |
| `lang/public_fr.php` (core) | `public:` | Frontend strings |

## Using i18n keys in squelettes

In squelette templates, use `#LANG` or the `|_T` filtre:

```html
[(#VAL{forum:aucun_message_forum}|_T)]
```

Or via the `<:module:cle:>` shorthand notation:

```html
<:forum:aucun_message_forum:>
<:myplugin:ma_cle{param=valeur}:>
```

---

## Common patterns

```php
// in verifier() — field error
$erreurs['titre'] = _T('info_obligatoire');   // 'info_obligatoire' is a core key (no module prefix → defaults to 'spip:')

// in traiter() — success message
return ['message_ok' => _T('ecrire:info_modification_enregistree')];

// conditional translation — use forced=false when optional
$label = _T('myplugin:libelle_optionnel', [], ['force' => false]) ?: 'Default';
```
